# Plan: Per-library diataxis docs

Allow libraries to provide tutorials, how-tos, and explanations that are built
and hosted in the charmlibs docs site alongside the existing site-global docs.

## Current state

- All diataxis docs are site-global, living directly in `.docs/`:
  - `tutorial.md` — a single leaf page (no toctree)
  - `how-to/index.md` — explicit toctree entries
  - `explanation/index.md` — explicit toctree entries
- Reference docs for packages are auto-generated from docstrings by the
  `package_docs.py` Sphinx extension into `reference/charmlibs/[interfaces/]`.
- Interface spec pages are generated by `interface_docs.py` into
  `reference/interfaces/`.
- Generated paths under `reference/` are in `.gitignore` and cleaned by `docs.just`.
- No library currently has a `docs/` directory.

## Convention: source layout in library directories

Each library can optionally provide a `docs/` directory:

```
<library>/docs/
├── tutorial.md               # single tutorial (optional)
├── how-to/                   # how-to guides (optional)
│   └── <name>.md
└── explanation/               # explanation pages (optional)
    └── <name>.md
```

Files can be `.md` or `.rst`. Reference docs remain auto-generated from
docstrings — they are not part of this feature.

Example:
```
interfaces/tls-certificates/docs/
├── tutorial.md
├── how-to/
│   └── integrate-as-provider.md
└── explanation/
    ├── certificate-renewal.md
    └── security.md
```

## Generated output at build time

The new extension copies library docs into `.docs/` under the matching diataxis
category, following the same `charmlibs/[interfaces/]` path convention used by
reference docs:

```
.docs/
├── tutorial.md                              # existing, unchanged
├── tutorials/                               # NEW section
│   ├── index.md                             # toctree: existing tutorial + globs
│   └── charmlibs/                           # GENERATED
│       ├── <general-lib>.md
│       └── interfaces/
│           └── <interface-lib>.md
├── how-to/
│   ├── index.md                             # existing, add glob toctree
│   └── charmlibs/                           # GENERATED
│       ├── <general-lib>/
│       │   └── <doc>.md
│       └── interfaces/
│           └── <interface-lib>/
│               └── <doc>.md
├── explanation/
│   ├── index.md                             # existing, add glob toctree
│   └── charmlibs/                           # GENERATED
│       ├── <general-lib>/
│       │   └── <doc>.md
│       └── interfaces/
│           └── <interface-lib>/
│               └── <doc>.md
```

Tutorials are single files (flat); how-to and explanation support multiple files
per library (one subdirectory per library).

## TOC structure

### Top-level restructure

The top-level toctree changes from singular "tutorial" to plural "tutorials":

```markdown
# before
tutorial
how-to/index
reference/index
explanation/index

# after
tutorials/index
how-to/index
reference/index
explanation/index
```

The existing `tutorial.md` stays at its current path and URL (`/charmlibs/tutorial/`).

### tutorials/index.md (new file)

```markdown
# Tutorials

``​`{toctree}
:maxdepth: 1

charmlibs: Write your first charm library </tutorial>
``​`

``​`{toctree}
:maxdepth: 1
:glob:

charmlibs/*
charmlibs/interfaces/*
``​`
```

The existing monorepo tutorial appears under `charmlibs:` (matching the existing
prefix convention for site-global docs about the monorepo). Library tutorials
are picked up by the glob.

### how-to/index.md (add glob toctree)

Append a second toctree block after the existing explicit entries:

```markdown
``​`{toctree}
:maxdepth: 1
:glob:

charmlibs/*/*
charmlibs/interfaces/*/*
``​`
```

### explanation/index.md (add glob toctree)

Same pattern as how-to.

### TOC entry naming

The extension prepends the library name to each copied file's H1 heading:

| Library type | Prefix | Example TOC entry |
|---|---|---|
| General library | `<lib-name>: ` | `pathops: Copy files to a container` |
| Interface library | `<interface-name>: ` | `tls-certificates: Certificate Renewal` |

This keeps entries identifiable while staying short enough to avoid wrapping in
the fixed-width sidebar. If the number of per-library docs grows large, the
flat list can later be reorganised into sub-TOCs under `charmlibs/` and
`charmlibs.interfaces/` headings.

### Task 1: New Sphinx extension `.docs/extensions/diataxis_docs.py`

On `builder-inited`:

1. Call `.scripts/ls.py packages --exclude-examples --exclude-placeholders --exclude-testing`
   to discover all library paths (same as `package_docs.py`).
2. For each library, check if `<root>/<lib>/docs/` exists.
3. For each diataxis category (`tutorial`, `how-to`, `explanation`):
   - If source files exist, read each file.
   - Determine the library name from the package path.
   - Prepend the library name to the first H1 heading (`# Title` → `# tls-certificates: Title`).
   - Rewrite relative non-HTTP links to absolute GitHub URLs (same pattern as
     `interface_docs.py` — `re.sub` on markdown link syntax).
   - Write to the output location using `_write_if_needed()`.
4. For tutorials: write to `tutorials/charmlibs/[interfaces/]<lib-name>.md`.
5. For how-to/explanation: write to `<category>/charmlibs/[interfaces/]<lib-name>/<doc>.md`.

The extension runs in every build (including per-package passes) since copying
markdown is cheap. This prevents toctree glob warnings during per-package builds.

Register in `conf.py` by adding `'diataxis_docs'` to `local_extensions`.

## Design decisions

| Decision | Rationale |
|---|---|
| Copy files, don't symlink | Allows modifying H1 headings and rewriting relative links portably |
| Prefix H1 with library name | Glob toctrees use the document's H1 as the TOC entry; prepending to H1 is the simplest way to get the desired prefix. Flat list for now; sub-TOCs later if needed |
| Rewrite relative links to GitHub URLs | Source docs live in library dirs; rendered docs are in `.docs/` — relative paths would break. Same pattern as `interface_docs.py` |
| Run in every build (including per-package) | Copying markdown is cheap; avoids toctree glob warnings in per-package builds |
| Keep `tutorial.md` at existing path | Preserves the existing URL; referenced from `tutorials/index.md` via absolute toctree path |
| Tutorials flat, how-to/explanation nested | Tutorials are single-file per library (flat under `tutorials/charmlibs/`); how-to/explanation support multiple files per library (need subdirectories) |
| Follow `charmlibs/[interfaces/]` path convention | Consistent with existing reference doc paths |

## Edge cases to handle

- **No libraries have docs yet**: glob toctrees match nothing. This should
  produce no warnings (Sphinx glob toctrees silently match empty), but validate
  during implementation. If it does warn, write a placeholder file (same pattern
  as `interface_docs.py`).
- **RST files**: the extension should handle both `.md` and `.rst` source files.
  H1 detection needs to handle both `# Title` (MyST) and RST underline syntax.
- **Nested relative links and images**: the link rewriting regex should handle
  both `[text](path)` markdown links and `.. image::` RST directives. Start
  with markdown-only (matching `interface_docs.py`) and extend if needed.

## Can this be one-shotted?

Yes — tasks 1–8 can be done in a single agent session. The TLS docs are
already committed (Task 0), so the agent has content to test with from the
start. The extension patterns are well-established in `package_docs.py` and
`interface_docs.py`, and the agent can validate iteratively with
`just docs html`. Task 8 (reformatting) is mechanical — fixing image refs,
links, and import paths in ~12 markdown files.

## Validation commands

- `just docs html interfaces/tls-certificates` — build the full docs, only
  including package reference docs from tls-certificates. This is the key
  integration test for the new extension machinery.
- `just docs ext-unit` — run unit tests for the Sphinx extensions.
- `just docs ext-static` — run pyright on the Sphinx extensions.
